9.1. Funktoren, Applikative Funktoren und Monaden🔗

9.1.1. Funktoren🔗

Konzeptionell kann man sich einen Funktor vorstellen als einen Container von Elementen zusammen mit einer Funktion, die die Elemente des Containers abbildet und dabei die Struktur des Containers erhält. Die Funktion wird meist map genannt.

Genau genommen ist ein Funktor also ein Typkonstruktor F : Type u → Type v zusammen mit einer Funktion map, so dass für eine Funktion f : α → β die Funktion map f den Typ F α → F β hat. Man sagt, dass map die Funktion f "liftet" (siehe Abbildung 9.1).

Nicht nur das: natürlich will man auch, dass map sich ordentlich verhält, d.h. mit der Identitätsfunktion id : α → α und der Komposition von Funktionen verträglich ist. Man nennt das auch Funktorialität oder die Gesetze des Funktors - mehr dazu nach den Beispielen.

9.1.1.1. Beispiele🔗

Das Standardbeispiel ist List, aber auch Option ist ein Funktor.

Es gibt auch einen Infix-Operator für map, nämlich <$> mit map f l = f <$> l, d.h. unsere Beispiele können wir auch so schreiben:

[4, 7]#eval String.length <$> ["lean", "haskell"] 
[4, 7]
["Lean", "Haskell"]#eval String.capitalize <$> ["lean", "haskell"] 
["Lean", "Haskell"]

Man kann f <$> l vielleicht so vorlesen: "f wird über l gemappt".

9.1.1.2. Gesetze für Funktoren🔗

Die Typklasse LawfulFunctor gibt die beiden Gesetze für Funktoren an:

  1. id <$> x = x, d.h. der Funktor erhält die Identitätsfunktion

  2. (h ∘ g) <$> x = h <$> g <$> x, d.h. der Funktor erhält die Komposition von Funktionen

Beispiele:

[1, 2, 3]
true#eval id <$> list_123 == list_123

[2, 8, 18]
[2, 8, 18]#eval (double ∘ square) <$> list_123 
[2, 8, 18]
true#eval double <$> square <$> list_123 == (double ∘ square) <$> list_123 
true
def option_5 := some 5 def option_n : Option Nat := none some 5#eval id <$> option_5 
some 5
none#eval id <$> option_n 
none
some 42#eval ((· + 2) ∘ (· * 8)) <$> option_5

some 42#eval (· + 2) <$> (· * 8) <$> option_5

none#eval ((· + 2) ∘ (· * 8)) <$> option_n

9.1.1.3. Übungen🔗

[09.1] Beweisen Sie, dass List die Gesetze des Funktors erfüllt. (Verwenden Sie dabei die Taktik simp nicht.)

[09.2] Beweisen Sie, dass Option die Gesetze des Funktors erfüllt. (Verwenden Sie dabei die Taktik simp nicht.)

Zusammenhang zur Kategorientheorie:

Wenn man die Kategorie LEAN mit den Typen als Objekte und den Funktionen zwischen den Typen als Morphismen nimmt, dann entspricht der Funktor in Lean dem Endofunktor in der Kategorientheorie. Dort definiert als:

Ein Funktor ist eine Abbildung von Kategorien (d.h. der Objekte und der Morphismen der Kategorie), die (1) den identischen Morphismus und (2) die Komposition von Morphismen erhält. Ein Endofunktor ist ein Funktor einer Kategorie auf sich selbst.

9.1.2. Applikative Funktoren🔗

Der Funktor liftet mit map eine Abbildung in den Funktortyp. Man kann sich aber auch den Fall vorstellen, dass der Funktortyp selbst einen Funktionstyp "enthält". Das führt zum applikativen Funktor, d.h. der Typklasse Applicative.

Wir haben in unserem Beispiel der Liste also nicht eine Funktion, sondern eine Liste von Funktionen und in unserem Beispiel der Option nicht eine Funktion, sondern eine optionale Funktion.

Das Konzept besteht darin, dass der applikative Funktor zwei Funktionen hat, die sich "ordentlich verhalten":

pure : α → F α
seq  : F (α → β) → F α → F β

Als Infix-Operator wird seq so geschrieben: <*>.

Die Funktion pure erlaubt es ein Objekt zu einem Objekt des Typs Applicative zu machen. Und mit seq kann man Funktionen verketten. Der Name "applikativer Funktor" kommt daher, dass man Funktionen "innerhalb" des Funktortyps anwendet.

Der applikative Funktor ist eine Verallgemeinerung des Funktors, in dem das Mapping verallgemeinert wird für Funktionen mit mehreren Argumenten. Stellen wir uns vor, wir wollten folgende Funktionen verwenden:

map0 : α → F α
map1 : (α → β) → F α → F β
map2 : (α → β → γ) → F α → F β → F γ
map3 : (α → β → γ → δ) → F α → F β → F γ → F δ

Dann wäre es nicht sehr geschickt, wenn wir jede dieser Funktionen definieren müssten. Tatsächlich kann man sie mit Applicative leicht bekommen:

map0 = pure

Mit f : α → β und x : F α ist

map1 f x = pure f <*> x

Mit f : α → β → γ, x : F α und y : F β ist

map2 f x y = pure f <*> x <*> y

usw.

Applicative hat auch noch zwei weitere Sequenzoperatoren SeqLeft (infix: <*) und SeqRight (infix: *>), die folgendermaßen funktionieren:

x <* y evaluiert x und y und hat als Ergebnis die Auswertung von x. Analog hat x *> y nach der Evaluierung von x und y den Wert von y.

Diese Operatoren werden viel in Parser-Bibliotheken verwendet.

9.1.2.2. Die Gesetze des applikativen Funktors🔗

Zunächst wollen wir prüfen, dass es sich tatsächlich um einen Funktor handelt. Das bedeutet, dass man map aus den beiden Funktionen pure und seq komponieren kann. Und wir haben das schon in Beispielen gesehen:

map (f : α → β) : F α → F β = seq (pure f) : F α → F β

Nun die Gesetze von Applicative:

pure id <*> v = v                                  -- Identität
pure f <*> pure x = pure (f x)                     -- Homomorphismus
u <*> pure y = pure (· y) <*> u                    -- Vertauschung
pure (· ∘ ·) <*> u <*> v <*> w = u <*> (v <*> w)  -- Komposition

9.1.2.3. Verzögerte Auswertung🔗

Im Code der Typklasse Seq hat die Funktion seq eine etwas andere Signatur als wir oben angegeben haben:

class Seq (f : Type u → Type v) where
  seq : {α β : Type u} → f (α → β) → (Unit → f α) → f β

Das bedeutet, dass seq gar nicht direkt F α als zweites Argument hat, sondern eine Funktion Unit → F α. Warum?

Lean wertet strikt (eager evaluation) aus. Möchte man jedoch verzögerte Auswertung (lazy evaluation), dann verwendet man einen sogenannten Thunk, eine Funktion Unit → α.

Wird eine Funktion f mit dem Argument x ausgewertet, dann wird zunächst x ausgewertet und dann der Wert als Argument an die Funktion f übergeben. Will man diese Auswertung nicht, dann verwendet man als Argument für f die Funktion fun _ => x. Da Funktionen Werte sind, ist dieses Argument also schon evaluiert.

Dieses Vorgehen ist bei seq sinnvoll, wie man am Beispiel von Option sieht. Dort ist seq so definiert:

seq f x :=
  match f with
  | none => none
  | some g => g <$> x () -- map

Im Fall none gibt es gar keine Funktion, die seq auf x anwenden könnte, deshalb wird x in seq f x in diesem Fall nicht ausgewertet.

some 4#eval some (· + 2) <*> some 2 
some 4
some 4#eval (· + 2) <$> some 2 
some 4
none#eval (none : Option (Nat → Nat)) <*> some 2 
none
none#eval some (· + 2) <*> none 
none

Beispiel für verzögerte Auswertung:

Nebenbei: Es gibt in Lean übrigens eine Struktur Thunk, die dafür sorgt, dass einmal ausgewertete Aufrufe der Funktion Unit → α vermerkt werden, so dass bei einem erneuten Aufruf die Auswertung nicht mehr nötig ist (siehe The Lean Language Reference: Lazy Computations).

Wenn man den Infix-Operator <*> verwendet, dann wird automatisch aus dem rechten Operanden eine Funktion der Unit gemacht:

9.1.3. Monaden🔗

Eine reine Funktion hängt nur von ihren Aufrufargumenten ab und ergibt für dieselben Argumentwerte stets das identische Ergebnis. Oft ist man aber in einer Situation, wo zusätzlich ein Berechnungskontext im Spiel ist: etwa ein Protokoll des Aufrufs der Funktion, eine Ausgabe oder der Zugriff auf einen globalen Zustand.

Wir reichern dazu eine reine Funktion f : α → β um diesen Berechnungskontext an und erhalten eine monadische Funktion f: α → m β. Der Typkonstruktor m wird dann als Monade bezeichnet.

Wir haben im vorherigen Kapitel schon gesehen, dass wir uns nun natürlich um die Komposition von solchen monadischen Funktionen kümmern müssen, wozu der Infix-Operator >=> dient. (Dieser Operator heißt auch "Fisch"-Operator wegen seines Aussehens oder Kleisli-Operator nach dem Schweizer Mathematiker Heinrich Kleisli, nach dem die Kleisli-Kategorie benannt ist, die dem Konzept der Komposition von monadischen Funktionen in der Kategorientheorie entspricht. Ich meine gehört zu haben, dass bei der Entwicklung des Konzepts der Monade diese zunächst als (Kleisli-)Tripel bezeichnet wurde.)

9.1.3.1. Die Typklasse Monad🔗

Die Monade wird in Lean (und auch in Haskell) über die Typklassenhierarchie definiert:

class Functor (f : Type u → Type v) : Type (max (u+1) v) where
  map : {α β : Type u} → (α → β) → f α → f β

class Pure (f : Type u → Type v) where
  pure {α : Type u} : α → f α

class Seq (f : Type u → Type v) : Type (max (u+1) v) where
  seq : {α β : Type u} → f (α → β) → (Unit → f α) → f β

class SeqLeft (f : Type u → Type v) : Type (max (u+1) v) where
  seqLeft : {α β : Type u} → f α → (Unit → f β) → f α

class SeqRight (f : Type u → Type v) : Type (max (u+1) v) where
  seqRight : {α β : Type u} → f α → (Unit → f β) → f β

class Applicative (f : Type u → Type v) extends
  Functor f, Pure f, Seq f, SeqLeft f, SeqRight f where
  map      := fun x y => Seq.seq (pure x) fun _ => y
  seqLeft  := fun a b => Seq.seq (Functor.map (Function.const _) a) b
  seqRight := fun a b => Seq.seq (Functor.map (Function.const _ id) a) b

class Bind (m : Type u → Type v) where
  bind : {α β : Type u} → m α → (α → m β) → m β

class Monad (m : Type u → Type v) extends Applicative m, Bind m :
  Type (max (u+1) v) where
  map      f x := bind x (Function.comp pure f)
  seq      f x := bind f fun y => Functor.map y (x ())
  seqLeft  x y := bind x fun a => bind (y ()) (fun _ => pure a)
  seqRight x y := bind x fun _ => y ()

Man kann hier gut sehen, wie man das Konzept der Typklasse softwaretechnisch einsetzt.

Man kann eine Monade auch definieren durch

m ist ein Funktor
pure : α → m α
join : m (m α) → m a

Dabei ist join definiert als

join (a : m (m α)) : m α := bind a id

9.1.3.2. Die Gesetze der Monade🔗

Wir haben die Gesetze der Monade schon im vorherigen Kapitel besprochen. Hier zur Erinnerung:

-- Linksidentität
pure x >>= g = g x
-- Rechtsidentität
m >>= pure = m
-- Assoziativität
(m >>= g) >>= h = m >>= (fun x => g x >>= h)

Beispiele für diese Gesetze mit Option:

def plus2 : Nat → Option Nat := fun n => some (n + 2) def mal3 : Nat → Option Nat := fun n => some (n * 3)

Linksidentität

some 3#eval pure 1 >>= plus2 
some 3
some 3#eval plus2 1 
some 3

Rechtsidentität

some 42#eval (some 42) >>= pure 
some 42

Assoziativität

some 9#eval ((some 1) >>= plus2) >>= mal3 
some 9
some 9#eval (some 1) >>= (fun x => plus2 x >>= mal3) 
some 9

some (some 42)#eval Option.map joino (some (some (some 42))) 
some (some 42)

map join macht aus einer Dreifachmonade eine Doppelmonade

some (some 42)#eval joino (some (some (some 42))) 
some (some 42)

joino auch

some 42#eval (joino ∘ Option.map joino) (some (some (some 42))) 
some 42
some 42#eval (joino ∘ joino) (some (some (some 42))) 
some 42

Es stellt sich heraus, dass in unserem Beispiel

join ∘ map join = join ∘ join

gilt, also gewissermaßen die Assoziativität des "Flachmachens".

Nun wollen wir am Beispiel sehen, wie join und map die Identität bilden können:

some 42#eval (joino ∘ Option.map pure) (some 42) 
some 42
some 42#eval (joino ∘ pure) (some 42) 
some 42

Sowohl join ∘ map pure als auch join ∘ pure sind die Identität.

Nun wollen wir noch sehen, ob sich join mit map "gut verträgt":

def add1 := (· + 1) some 41#eval joino (some (some 41)) 
some 41
some 42#eval Option.map add1 (some 41) 
some 42
some 42#eval (Option.map add1 ∘ joino) (some (some 41)) 
some 42
some (some 42)#eval Option.map (Option.map add1) (some (some 41)) 
some (some 42)
some 42#eval joino (some (some 42)) 
some 42
some 42#eval (joino ∘ Option.map (Option.map add1)) (some (some 41)) 
some 42

Am Beispiel sehen wir, dass

join ∘ map (map f) = map f ∘ join

erfüllt ist.

Dies ist natürlich nicht nur in unserem Beispiel so, sondern gilt allgemein.

Mit map, pure und join können wir die Gesetze der Monade so ausdrücken:

join ∘ map join = join ∘ join
join ∘ map pure = join ∘ pure = id
join ∘ map (map f) = map f ∘ join

Dies erläutert die Verbindung zur mathematischen Definition der Monade:

Eine Monade T auf der Kategorie C ist ein Endofunktor T: C \to C zusammen mit zwei natürlichen Transformationen \eta: 1_C \to T und \mu: T ∘ T \to T, die folgende Eigenschaften erfüllen:

  • \mu ∘ T\mu = \mu ∘ \mu T (als natürliche Transformationen T^3 \to T)

  • \mu ∘ T\eta = \mu ∘ \eta T = 1_T (als natürliche Transformationen T \to T)

Ein Monoid ist eine algebraische Struktur mit einer assoziativen binären Verknüpfung und einem neutralen Element.

Die Endofunktoren T mnnehmen wir als Elemente einer algebraischen Struktur. Man kann dann \mu als binäre Verknüpfung sehen und \eta als neutrales Element. Und die beiden Eigenschaften machen diese algebraische Struktur zu einem Monoid.

So verstehen wir wie in der Kategorientheorie die Monade definiert ist, siehe Wikipedia:

The first axiom is akin to the associativity in monoids if we think of \mu as the monoid's binary operation, and the second axiom is akin to the existence of an identity element (which we think of as given by \eta). Indeed, a monad on C can alternatively be defined as a monoid in the category \mathbf{End}_{C} whose objects are the endofunctors of C and whose morphisms are the natural transformations between them, with the monoidal structure induced by the composition of endofunctors.

Dabei entspricht \eta : 1_C \to T gerade pure und \mu : T^2 \to T entspricht join.

join ∘ map join = join ∘ join verlangt die Assoziativität: Hat man 3 Schichten von Monaden, dann ist es egal, ob man erst die äußere oder die innere Schicht mit join flach macht oder umgekehrt.

join ∘ map pure = join ∘ pure = id bedeutet, dass startend mit einer Schicht der Monade es egal ist, ob man eine weitere Schicht innen oder außen bildet und dann erneut flach macht.

Und schließlich sagt join ∘ map (map f) = map f ∘ join, dass join eine natürliche Transformation ist.

Dieses Gesetz ist allerdings "for free", weil es wegen parametrischem Polymorphismus aus dem Typ von join folgt: siehe Barrtosz Milewski: Money for Nothing and Theorems for Free

Erläuterung:

Unter einer natürlichen Transformation versteht man eine strukturerhaltende Abbildung von Funktoren. Genauer:

Seien C und D Kategorien und F und G Funktoren von C nach D. Eine natürliche Transformation μ: F → G ist eine Funktion, die jedem Objekt \alpha einen Morphismus μ_\alpha zuordnet, so dass für alle f : \alpha \to \beta in C das folgende Diagramm kommutiert (das heißt, dass die Abbildung eines Objekts erst nach unten und dann rechts identisch ist mit erst nach rechts und dann nach unten):

In unserer Situation handelt es sich um die Kategorie LEAN, deren Objekte Typen und deren Morphismen Funktionen sind. Eine Monade ist ein Funktor (mit den genannten zusätzlichen Eigenschaften), den wir in folgender Abbildung als $T$ bezeichnen. Gegeben sei eine Funktion f : \alpha \to \alpha, sowie join, die Funktion join : T (T \alpha) \to T \alpha.

Die Eigenschaft join ∘ map (map f) = map f ∘ join können wir dann durch das kommutierende Diagramm in Abb. 9.3 ausdrücken.

Daran sehen wir unmittelbar, dass es sich bei join um eine natürliche Transformation von T ∘ T \to T handelt.

Tatsächlich muss man dieses Gesetz für Monaden nicht beweisen, denn es ergibt sich aus dem Typ von join. Dazu überlegt man:

Wenn wir für einen beliebigen Typ α eine Funktion j : F α → G α haben und F und G sind Funktoren, dann muss j ∘ map f = map f ∘ j gelten, denn dies folgt daraus, dass wir einen beliebigen Typ α haben

  • parametrischer Polymorphismus hat zur Folge, dass j eine natürliche Transformation sein muss.

Setzen wir nun j als join, G als T und F als T ∘ T, dann ergibt sich join ∘ map (map f) = map ∘ join. Das bedeutet, dass sich aus dem Typ von join diese Eigenschaft ergibt. Deshalb spricht man von einem "theorem for free".
