4.3. Applikative Funktoren🔗

Der Funktor liftet mit map eine Abbildung in den Funktortyp. Man kann sich aber auch den Fall vorstellen, dass der Funktortyp selbst einen Funktionstyp „enthält". Das führt zum applikativen Funktor, d.h. der Typklasse Applicative.

Das Konzept besteht darin, dass der applikative Funktor zwei Funktionen hat, die sich „ordentlich verhalten":

pure : α → F α
seq  : F (α → β) → F α → F β

Als Infix-Operator wird seq so geschrieben: <*>. Die Funktion pure erlaubt es ein Objekt zu einem Objekt des Typs Applicative zu machen. Und mit seq kann man Funktionen verketten. Der Name „applikativer Funktor" kommt daher, dass man Funktionen „innerhalb" des Funktortyps anwendet.

Der applikative Funktor ist eine Verallgemeinerung des Funktors, in dem das Mapping verallgemeinert wird für Funktionen mit mehreren Argumenten. Stellen wir uns vor, wir wollten folgende Funktionen verwenden:

map0 : α → F α
map1 : (α → β) → F α → F β
map2 : (α → β → γ) → F α → F β → F γ
map3 : (α → β → γ → δ) → F α → F β → F γ → F δ

Tatsächlich kann man sie mit Applicative leicht bekommen: map0 = pure, map1 f x = pure f <*> x, map2 f x y = pure f <*> x <*> y usw.

4.3.1. Beispiele🔗

Wir sehen uns wieder Beispiele mit List und Option an.

Beispiel List — wir können eine Liste von Funktionen auf eine Liste von Werten durch seq anwenden:

[3, 4, 5]#eval [(· + 2)] <*> [1, 2, 3] 
[3, 4, 5]
[3, 4, 5]#eval (· + 2) <$> [1, 2, 3] 
[3, 4, 5]
[4, 5, 6, 5, 6, 7, 6, 7, 8]#eval [(· + ·)] <*> [1, 2, 3] <*> [3, 4, 5] 
[4, 5, 6, 5, 6, 7, 6, 7, 8]

List.seq wendet jede Funktion der Liste auf jedes Element der zweiten Liste an. Das Resultat ist eine Liste unärer Funktionen der Art add_n für jedes Element n der zweiten Liste. Nun wird jede dieser Funktionen auf jedes Element der dritten Liste angewandt.

[3, 4, 5, 5, 10, 15]#eval [(· + 2), (· * 5)] <*> [1, 2, 3] 
[3, 4, 5, 5, 10, 15]
[3, 4, 5, 5, 10, 15, 0, 1, 2]#eval [(· + 2), (· * 5), (· - 1)] <*> [1, 2, 3] 
[3, 4, 5, 5, 10, 15, 0, 1, 2]

Man sieht, dass jede der Funktionen auf jeden Wert der Liste angewendet wird. Damit kann man auch ein bisschen Kombinatorik betreiben:

[(0, 0), (0, 1), (0, 2), (1, 0), (1, 1), (1, 2)]#eval Prod.mk <$> (List.range 2) <*> (List.range 3) 
[(0, 0), (0, 1), (0, 2), (1, 0), (1, 1), (1, 2)]
[(0, 0), (0, 1), (0, 2), (1, 0), (1, 1), (1, 2)]#eval (· , ·) <$> (List.range 2) <*> (List.range 3) 
[(0, 0), (0, 1), (0, 2), (1, 0), (1, 1), (1, 2)]
[(1, 3, 5), (1, 4, 5), (2, 3, 5), (2, 4, 5)]#eval (· , · , ·) <$> [1, 2] <*> [3, 4] <*> [5] 
[(1, 3, 5), (1, 4, 5), (2, 3, 5), (2, 4, 5)]

Beispiel Option:

some 42#eval pure (· * ·) <*> some 2 <*> some 21 
some 42
some 42#eval (· * ·) <$> some 21 <*> some 2 
some 42
none#eval pure (· * ·) <*> some 2 <*> none 
none
none#eval pure (· * ·) <*> none <*> some 21 
none

Applicative hat auch noch zwei weitere Sequenzoperatoren SeqLeft (infix: <*) und SeqRight (infix: *>): x <* y evaluiert x und y und hat als Ergebnis die Auswertung von x; analog hat x *> y nach der Evaluierung von x und y den Wert von y. Diese Operatoren werden viel in Parser-Bibliotheken verwendet.

4.3.2. Die Gesetze des applikativen Funktors🔗

Zunächst wollen wir prüfen, dass es sich tatsächlich um einen Funktor handelt. Das bedeutet, dass man map aus den beiden Funktionen pure und seq komponieren kann:

map (f : α → β) : F α → F β = seq (pure f) : F α → F β

Nun die Gesetze von Applicative:

pure id <*> v = v                                  -- Identität
pure f <*> pure x = pure (f x)                     -- Homomorphismus
u <*> pure y = pure (· y) <*> u                    -- Vertauschung
pure (· ∘ ·) <*> u <*> v <*> w = u <*> (v <*> w)  -- Komposition

Beispiel List:

[1, 2, 3]#eval pure id <*> [1, 2, 3] 
[1, 2, 3]
def f : Nat → Nat := (· + 2) [3]#eval (pure f : List (Nat → Nat)) <*> pure 1 
[3]
[3]#eval (pure (f 1) : List Nat) 
[3]
true#eval pure f <*> pure 1 == (pure (f 1) : List Nat) 
true
def g : List (Nat → Nat) := [(· + 2)] [42]#eval g <*> pure 40 
[42]
[42]#eval pure (· 40) <*> g 
[42]
true#eval g <*> pure 40 == pure (· 40) <*> g 
true

42

[10, 11]
[10, 11]#eval u <*> (v <*> w)

4.3.3. Verzögerte Auswertung🔗

Im Code der Typklasse Seq hat die Funktion seq eine etwas andere Signatur als wir oben angegeben haben:

class Seq (f : Type u → Type v) where
  seq : {α β : Type u} → f (α → β) → (Unit → f α) → f β

Das bedeutet, dass seq gar nicht direkt F α als zweites Argument hat, sondern eine Funktion Unit → F α. Lean wertet strikt (eager evaluation) aus. Möchte man jedoch verzögerte Auswertung (lazy evaluation), dann verwendet man einen sogenannten Thunk, eine Funktion Unit → α.

Dieses Vorgehen ist bei seq sinnvoll, wie man am Beispiel von Option sieht:

seq f x :=
  match f with
  | none => none
  | some g => g <$> x ()

Im Fall none gibt es gar keine Funktion, die seq auf x anwenden könnte, deshalb wird x in seq f x in diesem Fall nicht ausgewertet.

some 4#eval some (· + 2) <*> some 2 
some 4
none#eval (none : Option (Nat → Nat)) <*> some 2 
none
none#eval some (· + 2) <*> none 
none

Beispiel für verzögerte Auswertung:

def MyDouble : Nat → Nat := fun n => 2 * n def MySquare (f : Unit → Nat) : Nat := f () * f () 32#eval MyDouble (MySquare (fun _ => MyDouble 2)) 
32
MyDouble (MySquare fun x => MyDouble 2) : ℕ#check MyDouble (MySquare (fun _ => MyDouble 2)) 
MyDouble (MySquare fun x => MyDouble 2) : ℕ
example : MyDouble (MySquare (fun _ => MyDouble 2)) = 32 := ⊢ MyDouble (MySquare fun x => MyDouble 2) = 32 ⊢ MyDouble ((fun x => MyDouble 2) () * (fun x => MyDouble 2) ()) = 32 ⊢ 2 * ((fun x => 2 * 2) () * (fun x => 2 * 2) ()) = 32 All goals completed! 🐙

Nebenbei: Es gibt in Lean übrigens eine Struktur Thunk, die dafür sorgt, dass einmal ausgewertete Aufrufe der Funktion Unit → α vermerkt werden, so dass bei einem erneuten Aufruf die Auswertung nicht mehr nötig ist (siehe The Lean Language Reference: Lazy Computations).

Wenn man den Infix-Operator <*> verwendet, dann wird automatisch aus dem rechten Operanden eine Funktion der Unit gemacht:

[fun x => x + 2] <*> [1, 2, 3] : List ℕ#check [(· + 2)] <*> [1, 2, 3] 
[fun x => x + 2] <*> [1, 2, 3] : List ℕ
