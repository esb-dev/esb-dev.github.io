7.2. Eigenschaften von Funktionen für Listen🔗

Die erste Eigenschaft, die wir zeigen wollen, sagt aus, dass die leere Liste die Länge 0 hat.

lemma length_nil : List.length ([] : List α) = 0 := α:Type u_1⊢ [].length = 0 All goals completed! 🐙

Die Funktion List.length ist folgendermaßen definiert:

def List.length : List α → Nat
  | nil       => 0
  | cons _ as => HAdd.hAdd (length as) 1

Die Taktik constructor verwendet den ersten passenden Konstruktor eines induktiv definierten Typs, bei List in unserem Beweis also nil (= []), und man erhält damit, dass die Länge der leeren Liste in List.length als 0 definiert wurde. (Siehe constructor.)

Wir hätten auch Folgendes tun können:

example : List.length ([] : List α) = 0 := α:Type u_1⊢ [].length = 0 All goals completed! 🐙

Die Taktik rfl beweist Ziele der Form ⊢ x = y, wenn x und y per Definition gleich sind (siehe rfl), was hier der Fall ist.

Folgendes Lemma sagt aus, dass eine Liste mit einem Element die Länge 1 hat. Beweisen Sie diese Aussage als Übung [07.1].

/-- warning: declaration uses 'sorry' -/ #guard_msgs in lemma length_singleton {α : Type} (a : α) : List.length [a] = 1 := α:Typea:α⊢ [a].length = 1 All goals completed! 🐙

Mit sorry kennzeichnen wir eine Beweisverpflichtung, die wir später noch einlösen wollen (und müssen). Wir können dann die Aussage zwar verwenden, aber sie wurde noch nicht bewiesen -- und Lean weiß das!

cons konstruiert eine neue Liste aus einem Element und der bisherigen Liste. Listen "wachsen" vorne, damit die bisherige Liste im Sinne des structure sharings ohne Kopieren für die neue Liste verwendet werden kann. Durch cons wird die Liste um 1 länger:

lemma length_cons : List.length (a :: as) = (List.length as) + 1 := α✝:Type u_1a:α✝as:List α✝⊢ (a :: as).length = as.length + 1 All goals completed! 🐙

Wenn man zwei Listen aneinander hängt, dann addieren sich ihre Längen. Der Beweis dafür ist etwas aufwändiger:

lemma length_append (as bs : List α) : List.length (as ++ bs) = List.length as + List.length bs := α:Type u_1as:List αbs:List α⊢ (as ++ bs).length = as.length + bs.length induction as with α:Type u_1bs:List α⊢ ([] ++ bs).length = [].length + bs.length -- simp α:Type u_1bs:List α⊢ ([] ++ bs).length = 0 + bs.length α:Type u_1bs:List α⊢ ([] ++ bs).length = bs.length All goals completed! 🐙 -- rewrite [List.nil_append] -- rfl α:Type u_1bs:List αa:αas:List αih:(as ++ bs).length = as.length + bs.length⊢ (a :: as ++ bs).length = (a :: as).length + bs.length -- simp [ih, Nat.succ_add] α:Type u_1bs:List αa:αas:List αih:(as ++ bs).length = as.length + bs.length⊢ (a :: (as ++ bs)).length = (a :: as).length + bs.length α:Type u_1bs:List αa:αas:List αih:(as ++ bs).length = as.length + bs.length⊢ (as ++ bs).length + 1 = (a :: as).length + bs.length α:Type u_1bs:List αa:αas:List αih:(as ++ bs).length = as.length + bs.length⊢ as.length + bs.length + 1 = (a :: as).length + bs.length α:Type u_1bs:List αa:αas:List αih:(as ++ bs).length = as.length + bs.length⊢ as.length + bs.length + 1 = as.length + 1 + bs.length All goals completed! 🐙

In diesem Beweis haben wir die Taktik induction eingesetzt, die einen Induktionsbeweis über die beiden Konstruktoren nil und cons von List durchführt. (Siehe induction.)

Die Taktik rw [h] setzt die (bereits bewiesene oder vorausgesetzte) Aussage h der Form h : x = y, also eine Gleichheitsaussage an allen passenden Stellen im Ziel ein (siehe rw). Zuerst verwenden wir als h das oben bewiesene Lemma length_nil. Danach das Theorem Nat.zero_add, das Lean bereits mitliefert. Danach List.nil_append.

Im Beweis werden die einzelnen Schritte durchgeführt, damit man sieht, was genau passiert. Wir sehen auch, dass rw nach einem rewrite automatisch rfl versucht (siehe rewrite).

Wie findet man bereits bewiesene Aussagen in Lean? Es gibt die Suchseite Loogle!. Geben wir dort zum Beispiel "zero", "add" ein, dann sind die beiden ersten Treffer Nat.add_zero und Nat.zero_add. A propos: Man kann daran leicht erkennen, welche Namenskonvention in Lean für solche Aussagen verwendet wird und deshalb kann man oft auch erraten, wie die gesuchte Aussage wohl heißen wird.

Im Beweis für den zweiten Konstruktor cons von List nennen wir die Induktionsvoraussetzung ih und wir müssen den Beweis so konstruieren, dass wir sie anwenden können. Dazu verwenden wir List.cons_append sowie length_cons. Dann sind wir fast fertig.

Die Taktik ac_rfl ist wie rfl jedoch zusätzlich modulo Assoziativ- und Kommutativgesetz (siehe ac_rfl). Sie erspart uns also mühseliges Anwenden von Nat.add_assoc und Nat.add_comm.

Es folgt die kurze Variante des Beweises, die die Taktik simp einsetzt:

example : List.length (as ++ bs) = List.length as + List.length bs := α✝:Type u_1as:List α✝bs:List α✝⊢ (as ++ bs).length = as.length + bs.length induction as with α✝:Type u_1bs:List α✝⊢ ([] ++ bs).length = [].length + bs.length All goals completed! 🐙 α✝:Type u_1bs:List α✝h:α✝as:List α✝ih:(as ++ bs).length = as.length + bs.length⊢ (h :: as ++ bs).length = (h :: as).length + bs.length α✝:Type u_1bs:List α✝h:α✝as:List α✝ih:(as ++ bs).length = as.length + bs.length⊢ as.length + bs.length + 1 = as.length + 1 + bs.length All goals completed! 🐙

Die Taktik simp (für simplifier) verwendet eine Datenbank von bewiesenen Aussagen und versucht das Ziel (oder mit simp at h auch die Voraussetzung h) durch Einsetzen dieser Aussagen zu vereinfachen (siehe simp). Man kann selbst eigene Aussagen dieser Datenbank hinzufügen, indem man das Lemma oder Theorem mit dem Attribut @simp versieht (Mehr über simp)Weil sich die Datenbank der Lemmata und Theoreme ändern kann (und wird), die simp verwendet, sollte man simp immer nur als letzte Taktik für ein Ziel einsetzen. Denn andernfalls könnte ein rewrite nach einem simp plötzlich scheitern, weil die Aussage des rewrite jetzt schon durch simp eingesetzt wird..

Noch kürzer wird der Beweis, wenn wir die Beweistaktik grind einsetzen, die viele Beweise automatisieren kann (siehe The Lean Language Reference: The grind tactic).

example : List.length (as ++ bs) = List.length as + List.length bs := α✝:Type u_1as:List α✝bs:List α✝⊢ (as ++ bs).length = as.length + bs.length All goals completed! 🐙

Mit grind kann man viele der folgenden Aussagen beweisen und muss sich wenig Gedanken über die Aussagen selbst machen. Dabei denkt der Mathematiker in mir, dass Beweise doch auch dazu beitragen sollten, den Sachverhalt besser zu verstehen.

Deshalb bleiben wir bei den ausführlicheren Beweisen und verwenden grind nur quasi zur Bestätigung.

lemma length_concat (as : List α) (a : α) : List.length (List.concat as a) = List.length as + 1 := α:Type u_1as:List αa:α⊢ (as.concat a).length = as.length + 1 cases as with α:Type u_1a:α⊢ ([].concat a).length = [].length + 1 All goals completed! 🐙 α:Type u_1a✝:αa:αas:List α⊢ ((a :: as).concat a✝).length = (a :: as).length + 1 All goals completed! 🐙

Die Taktik cases zerlegt das Beweisziel in Teilziele pro Konstruktor des Typs des Terms, hier für die beiden Konstruktoren von List. cases ist ähnlich zu induction, erstellt aber keine Induktionsvoraussetzung (siehe cases).

example (as : List α) (a : α) : List.length (List.concat as a) = List.length as + 1 := α:Type u_1as:List αa:α⊢ (as.concat a).length = as.length + 1 All goals completed! 🐙

Nun wollen wir noch einige Eigenschaften von map beweisen. Die Funktion List.map wendet eine unäre Funktion auf jedes Element einer Liste an:

def map (f : α → β) : List α → List β
  | []      => []
  | a :: as => f a :: map f as

Wenn wir mittels der Identitätsfunktion "mappen", dann erhalten wir die Eingabeliste als Ergebnis:

lemma map_id {α : Type} (xs : List α) : List.map (fun x => x) xs = xs := α:Typexs:List α⊢ List.map (fun x => x) xs = xs α:Type⊢ List.map (fun x => x) [] = []α:Typehead✝:αtail✝:List α⊢ List.map (fun x => x) (head✝ :: tail✝) = head✝ :: tail✝ α:Type⊢ List.map (fun x => x) [] = [] All goals completed! 🐙 α:Typehead✝:αtail✝:List α⊢ List.map (fun x => x) (head✝ :: tail✝) = head✝ :: tail✝ All goals completed! 🐙

Der Aufzählungspunkt steht je ein Ziel, das durch cases entstanden ist, siehe Focussing.

example {α : Type} (xs : List α) : List.map (fun x => x) xs = xs := α:Typexs:List α⊢ List.map (fun x => x) xs = xs All goals completed! 🐙

Beweisen Sie als Übungsaufgabe [07.2], dass map und append vertauschbar sind:

/-- warning: declaration uses 'sorry' -/ #guard_msgs in lemma map_append {α β : Type} (f : α → β) (xs ys : List α) : List.map f (xs ++ ys) = List.map f xs ++ List.map f ys := α:Typeβ:Typef:α → βxs:List αys:List α⊢ List.map f (xs ++ ys) = List.map f xs ++ List.map f ys All goals completed! 🐙

In Abschnitt 7.4 dieses Kapitels werden wir uns mit dem MergeSort befassen. Wir werden die Funktion mergeSort entwickeln und wollen dann beweisen, dass diese Funktion terminiert. Dafür werden wir einige Aussagen über Listen benötigen. Diese Aussagen wollen wir nun im Vorgriff beweisen.

Die Funktion split teilt eine Liste in der Mitte. Sie verwendet dazu die Listenfunktionen take und drop, die wir schon in Kapitel 4 kennengelernt haben.

Wir wollen nun beweisen, dass beide Teile nach dem Teilen durch split kleiner sind als die Liste selbst, vorausgesetzt, sie hatte mindestens 2 Elemente.

Dazu beweisen wir zunächst zwei Aussagen über (n + 1)/2:

lemma split_lt_n (n : Nat) : n ≥ 2 → (n + 1)/2 < n := n:ℕ⊢ n ≥ 2 → (n + 1) / 2 < n ⊢ 0 ≥ 2 → (0 + 1) / 2 < 0n✝:ℕ⊢ n✝ + 1 ≥ 2 → (n✝ + 1 + 1) / 2 < n✝ + 1 ⊢ 0 ≥ 2 → (0 + 1) / 2 < 0 All goals completed! 🐙 n✝:ℕ⊢ n✝ + 1 ≥ 2 → (n✝ + 1 + 1) / 2 < n✝ + 1 n✝:ℕh:n✝ + 1 ≥ 2⊢ (n✝ + 1 + 1) / 2 < n✝ + 1 n✝:ℕh:n✝ + 1 ≥ 2⊢ n✝ + 1 + 1 < (n✝ + 1) * 2n✝:ℕh:n✝ + 1 ≥ 2⊢ 0 < 2 n✝:ℕh:n✝ + 1 ≥ 2⊢ n✝ + 1 + 1 < (n✝ + 1) * 2 n✝:ℕh:n✝ + 1 ≥ 2⊢ n✝ + 1 + 1 < n✝ * 2 + 1 * 2 All goals completed! 🐙 n✝:ℕh:n✝ + 1 ≥ 2⊢ 0 < 2 All goals completed! 🐙

Im Fall n = 0 können wir die Taktik trivial anwenden, denn die Voraussetzung ist gar nicht erfüllt. trivial "erkennt" diesen Widerspruch in den Voraussetzungen.

Im Fall cons führt intro h die linke Seite der Implikation n + 1 ≥ 2 als Voraussetzung (hypotheses) ein. Nun verwenden wir die Aussage

Nat.div_lt_iff_lt_mul {k x y : ℕ} (Hk : 0 < k) : x / k < y ↔ x < y * k#check Nat.div_lt_iff_lt_mul 
Nat.div_lt_iff_lt_mul {k x y : ℕ} (Hk : 0 < k) : x / k < y ↔ x < y * k

wodurch wir zwei Beweisverpflichtungen erhalten, nämlich die Voraussetzungen von Nat.div_lt_iff_lt_mul:

n + 1 + 1 < (n + 1) * 2
0 < 2

Die zweite ist trivial, für die erste setzen wir das mächtige Geschütz der Taktik linarith ein, die lineare Gleichungen und Ungleichungen für die natürlichen Zahlen oft lösen kann.

example (n : ℕ) : n ≥ 2 → (n + 1)/2 < n := n:ℕ⊢ n ≥ 2 → (n + 1) / 2 < n All goals completed! 🐙

Nun beweisen wir die zweite Aussage über (n + 1)/2:

In diesen beiden Beweisen kommen folgende Taktiken vor, die wir bisher nicht kennengelernt haben:

  • have führt eine neue Aussage ein, zusammen mit einer Beweisverpflichtung dafür. Erbringt man den Beweis, dann wird die Aussage in den Stack der Voraussetzungen übernommen, die man im Folgenden verwenden kann. (Siehe have.)

  • exact kann man verwenden, wenn das Beweisziel exakt mit einer Voraussetzung übereinstimmt (siehe exact).

  • apply wendet eine Funktion an (siehe apply). Zum Beispiel ist split_lt_n (n : Nat) : n ≥ 2 → (n + 1) / 2 < n eine Funktion mit der Aussage n ≥ 2 als Argument. Deshalb können wir sie auf das Ziel anwenden.

Damit beenden wir den Abschnitt über Listen. Die beiden zuletzt bewiesenen Aussagen werden später beim MergeSort verwendet.
