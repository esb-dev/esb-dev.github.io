4.1. Funktoren, Applikative Funktoren und Monaden🔗

4.1.1. Funktoren🔗

Konzeptionell kann man sich einen Funktor vorstellen als einen Container von Elementen zusammen mit einer Funktion, die die Elemente des Containers abbildet und dabei die Struktur des Containers erhält. Die Funktion wird meist map genannt.

Genau genommen ist ein Funktor also ein Typkonstruktor F : Type u → Type v zusammen mit einer Funktion map, so dass für eine Funktion f : α → β die Funktion map f den Typ F α → F β hat. Man sagt, dass map die Funktion f „liftet".

Nicht nur das: natürlich will man auch, dass map sich ordentlich verhält, d.h. mit der Identitätsfunktion id : α → α und der Komposition von Funktionen verträglich ist. Man nennt das auch Funktorialität oder die Gesetze des Funktors.

4.1.1.1. Beispiele🔗

Das Standardbeispiel ist List, aber auch Option ist ein Funktor.

Es gibt auch einen Infix-Operator für map, nämlich <$> mit map f l = f <$> l, d.h. unsere Beispiele können wir auch so schreiben:

[2, 2, 2]#eval (fun _ => 2) <$> [1, 2, 3] 
[2, 2, 2]
["1", "2", "3"]#eval (fun x => toString x) <$> [1, 2, 3] 
["1", "2", "3"]
[4, 7]#eval (fun s => s.length) <$> ["lean", "haskell"] 
[4, 7]
[4, 7]#eval String.length <$> ["lean", "haskell"] 
[4, 7]
["Lean", "Haskell"]#eval String.capitalize <$> ["lean", "haskell"] 
["Lean", "Haskell"]

Man kann f <$> l vielleicht so vorlesen: „f wird über l gemappt".

4.1.1.2. Gesetze für Funktoren🔗

Die Typklasse LawfulFunctor gibt die beiden Gesetze für Funktoren an: id <$> x = x (der Funktor erhält die Identitätsfunktion) und (h ∘ g) <$> x = h <$> g <$> x (der Funktor erhält die Komposition von Funktionen).

Beispiele:

[1, 2, 3]

true

[2, 8, 18]
[2, 8, 18]#eval (double ∘ square) <$> list_123 
[2, 8, 18]

true

none#eval id <$> option_n

4.1.4. Applikative Funktoren🔗

Der Funktor liftet mit map eine Abbildung in den Funktortyp. Man kann sich aber auch den Fall vorstellen, dass der Funktortyp selbst einen Funktionstyp „enthält". Das führt zum applikativen Funktor, d.h. der Typklasse Applicative.

Das Konzept besteht darin, dass der applikative Funktor zwei Funktionen hat, die sich „ordentlich verhalten":

pure : α → F α
seq  : F (α → β) → F α → F β

Als Infix-Operator wird seq so geschrieben: <*>. Die Funktion pure erlaubt es ein Objekt zu einem Objekt des Typs Applicative zu machen. Und mit seq kann man Funktionen verketten. Der Name „applikativer Funktor" kommt daher, dass man Funktionen „innerhalb" des Funktortyps anwendet.

Der applikative Funktor ist eine Verallgemeinerung des Funktors, in dem das Mapping verallgemeinert wird für Funktionen mit mehreren Argumenten. Stellen wir uns vor, wir wollten folgende Funktionen verwenden:

map0 : α → F α
map1 : (α → β) → F α → F β
map2 : (α → β → γ) → F α → F β → F γ
map3 : (α → β → γ → δ) → F α → F β → F γ → F δ

Tatsächlich kann man sie mit Applicative leicht bekommen: map0 = pure, map1 f x = pure f <*> x, map2 f x y = pure f <*> x <*> y usw.

4.1.4.1. Beispiele🔗

Wir sehen uns wieder Beispiele mit List und Option an.

Beispiel List — wir können eine Liste von Funktionen auf eine Liste von Werten durch seq anwenden:

[3, 4, 5]#eval [(· + 2)] <*> [1, 2, 3] 
[3, 4, 5]
[3, 4, 5]#eval (· + 2) <$> [1, 2, 3] 
[3, 4, 5]
[4, 5, 6, 5, 6, 7, 6, 7, 8]#eval [(· + ·)] <*> [1, 2, 3] <*> [3, 4, 5]

List.seq wendet jede Funktion der Liste auf jedes Element der zweiten Liste an. Das Resultat ist eine Liste unärer Funktionen der Art add_n für jedes Element n der zweiten Liste. Nun wird jede dieser Funktionen auf jedes Element der dritten Liste angewandt.

[3, 4, 5, 5, 10, 15]#eval [(· + 2), (· * 5)] <*> [1, 2, 3] 
[3, 4, 5, 5, 10, 15]

[3, 4, 5, 5, 10, 15, 0, 1, 2]

Man sieht, dass jede der Funktionen auf jeden Wert der Liste angewendet wird. Damit kann man auch ein bisschen Kombinatorik betreiben:

Applicative hat auch noch zwei weitere Sequenzoperatoren SeqLeft (infix: <*) und SeqRight (infix: *>): x <* y evaluiert x und y und hat als Ergebnis die Auswertung von x; analog hat x *> y nach der Evaluierung von x und y den Wert von y. Diese Operatoren werden viel in Parser-Bibliotheken verwendet.

4.1.4.2. Die Gesetze des applikativen Funktors🔗

Zunächst wollen wir prüfen, dass es sich tatsächlich um einen Funktor handelt. Das bedeutet, dass man map aus den beiden Funktionen pure und seq komponieren kann:

map (f : α → β) : F α → F β = seq (pure f) : F α → F β

Nun die Gesetze von Applicative:

pure id <*> v = v                                  -- Identität
pure f <*> pure x = pure (f x)                     -- Homomorphismus
u <*> pure y = pure (· y) <*> u                    -- Vertauschung
pure (· ∘ ·) <*> u <*> v <*> w = u <*> (v <*> w)  -- Komposition

4.1.4.3. Verzögerte Auswertung🔗

Im Code der Typklasse Seq hat die Funktion seq eine etwas andere Signatur als wir oben angegeben haben:

class Seq (f : Type u → Type v) where
  seq : {α β : Type u} → f (α → β) → (Unit → f α) → f β

Das bedeutet, dass seq gar nicht direkt F α als zweites Argument hat, sondern eine Funktion Unit → F α. Lean wertet strikt (eager evaluation) aus. Möchte man jedoch verzögerte Auswertung (lazy evaluation), dann verwendet man einen sogenannten Thunk, eine Funktion Unit → α.

Dieses Vorgehen ist bei seq sinnvoll, wie man am Beispiel von Option sieht:

seq f x :=
  match f with
  | none => none
  | some g => g <$> x ()

Im Fall none gibt es gar keine Funktion, die seq auf x anwenden könnte, deshalb wird x in seq f x in diesem Fall nicht ausgewertet.

some 4#eval some (· + 2) <*> some 2 
some 4
none#eval (none : Option (Nat → Nat)) <*> some 2 
none
none#eval some (· + 2) <*> none 
none

Beispiel für verzögerte Auswertung:

def MyDouble : Nat → Nat := fun n => 2 * n def MySquare (f : Unit → Nat) : Nat := f () * f () 32#eval MyDouble (MySquare (fun _ => MyDouble 2)) 
32
MyDouble (MySquare fun x => MyDouble 2) : ℕ#check MyDouble (MySquare (fun _ => MyDouble 2)) 
MyDouble (MySquare fun x => MyDouble 2) : ℕ
example : MyDouble (MySquare (fun _ => MyDouble 2)) = 32 := ⊢ MyDouble (MySquare fun x => MyDouble 2) = 32 ⊢ MyDouble ((fun x => MyDouble 2) () * (fun x => MyDouble 2) ()) = 32 ⊢ 2 * ((fun x => 2 * 2) () * (fun x => 2 * 2) ()) = 32 All goals completed! 🐙

Nebenbei: Es gibt in Lean übrigens eine Struktur Thunk, die dafür sorgt, dass einmal ausgewertete Aufrufe der Funktion Unit → α vermerkt werden, so dass bei einem erneuten Aufruf die Auswertung nicht mehr nötig ist (siehe The Lean Language Reference: Lazy Computations).

Wenn man den Infix-Operator <*> verwendet, dann wird automatisch aus dem rechten Operanden eine Funktion der Unit gemacht:

[fun x => x + 2] <*> [1, 2, 3] : List ℕ#check [(· + 2)] <*> [1, 2, 3]

4.1.5. Monaden🔗

Eine reine Funktion hängt nur von ihren Aufrufargumenten ab und ergibt für dieselben Argumentwerte stets das identische Ergebnis. Oft ist man aber in einer Situation, wo zusätzlich ein Berechnungskontext im Spiel ist: etwa ein Protokoll des Aufrufs der Funktion, eine Ausgabe oder der Zugriff auf einen globalen Zustand. Wir reichern dazu eine reine Funktion f : α → β um diesen Berechnungskontext an und erhalten eine monadische Funktion f : α → m β. Der Typkonstruktor m wird dann als Monade bezeichnet.

4.1.5.1. Die Typklasse Monad🔗

Die Monade wird in Lean (und auch in Haskell) über die Typklassenhierarchie definiert:

class Functor (f : Type u → Type v) : Type (max (u+1) v) where
  map : {α β : Type u} → (α → β) → f α → f β

class Pure (f : Type u → Type v) where
  pure {α : Type u} : α → f α

class Seq (f : Type u → Type v) : Type (max (u+1) v) where
  seq : {α β : Type u} → f (α → β) → (Unit → f α) → f β

class Applicative (f : Type u → Type v) extends
    Functor f, Pure f, Seq f, SeqLeft f, SeqRight f where
  map      := fun x y => Seq.seq (pure x) fun _ => y
  seqLeft  := fun a b => Seq.seq (Functor.map (Function.const _) a) b
  seqRight := fun a b =>
    Seq.seq (Functor.map (Function.const _ id) a) b

class Bind (m : Type u → Type v) where
  bind : {α β : Type u} → m α → (α → m β) → m β

class Monad (m : Type u → Type v) extends
    Applicative m, Bind m : Type (max (u+1) v) where
  map      f x := bind x (Function.comp pure f)
  seq      f x := bind f fun y => Functor.map y (x ())
  seqLeft  x y := bind x fun a => bind (y ()) (fun _ => pure a)
  seqRight x y := bind x fun _ => y ()

Man kann eine Monade auch definieren durch einen Funktor m, pure : α → m α und join : m (m α) → m a, wobei join definiert ist als:

join (a : m (m α)) : m α := bind a id

4.1.5.2. Die Gesetze der Monade🔗

Hier zur Erinnerung die Gesetze der Monade:

pure x >>= g = g x                              -- Linksidentität
m >>= pure = m                                  -- Rechtsidentität
(m >>= g) >>= h = m >>= (fun x => g x >>= h)   -- Assoziativität

Beispiele für diese Gesetze mit Option:

def plus2 : Nat → Option Nat := fun n => some (n + 2) def mal3 : Nat → Option Nat := fun n => some (n * 3) some 3#eval pure 1 >>= plus2 
some 3
some 3#eval plus2 1 
some 3
some 42#eval (some 42) >>= pure 
some 42
some 9#eval ((some 1) >>= plus2) >>= mal3 
some 9
some 9#eval (some 1) >>= (fun x => plus2 x >>= mal3) 
some 9

Zum Abschluss dieses Abschnitts über Monaden wollen wir uns die alternative Definition einer Monade mit join etwas genauer ansehen.

Wir beginnen mit Beispielen der Monade Option. Wir definieren die Funktion join (hier speziell joino für Options). join hat als Argument eine „Doppelmonade" und macht sie „flach":

def joino (o2 : Option (Option α)) : Option α := bind o2 id @joino : {α : Type u_1} → Option (Option α) → Option α#check @joino 
@joino : {α : Type u_1} → Option (Option α) → Option α
some 42#eval joino (some (some 42)) 
some 42
some (some 42)#eval joino (some (some (some 42))) 
some (some 42)

Wir wollen nun überprüfen wie es sich verhält, wenn wir bei einer „Dreifachmonade" erst die innere Schicht und dann die äußere mit join „flach" machen oder umgekehrt:

some 42#eval (joino ∘ Option.map joino) (some (some (some 42))) 
some 42
some 42#eval (joino ∘ joino) (some (some (some 42))) 
some 42

Es stellt sich heraus, dass join ∘ map join = join ∘ join gilt, also gewissermaßen die Assoziativität des „Flachmachens".

Nun wollen wir am Beispiel sehen, wie join und map die Identität bilden können:

some 42#eval (joino ∘ Option.map pure) (some 42) 
some 42
some 42#eval (joino ∘ pure) (some 42) 
some 42

Sowohl join ∘ map pure als auch join ∘ pure sind die Identität.

Nun wollen wir noch sehen, ob sich join mit map „gut verträgt":

def add1 := (· + 1) some 42#eval (Option.map add1 ∘ joino) (some (some 41)) 
some 42
some 42#eval (joino ∘ Option.map (Option.map add1)) (some (some 41)) 
some 42

Am Beispiel sehen wir, dass join ∘ map (map f) = map f ∘ join erfüllt ist. Mit map, pure und join können wir die Gesetze der Monade so ausdrücken:

join ∘ map join = join ∘ join
join ∘ map pure = join ∘ pure = id
join ∘ map (map f) = map f ∘ join

Dabei entspricht $eta : 1C to T$ gerade pure und $mu : T^2 to T$ entspricht join. Die mathematische Definition der Monade lautet: Eine Monade $T$ auf der Kategorie $C$ ist ein Endofunktor $T: C to C$ zusammen mit zwei natürlichen Transformationen $eta: 1C to T$ und $mu: T circ T to T$. Wenn Sie an der mathematischen Definition der Monade interessiert sind, empfehle ich Bartosz Milewski: Category Theory for Programmers.
